# Changelog

Per-release behavior changes for the `omop-pipeline-builder` skill. Entries land here when each release ships to this public repo. The first release entry is v2.0.7 — earlier versions (v1.1 through v2.0.6) were development versions with no customer distribution.

---

## v2.0.7

**Headline capability — source-to-concept mapping draft generator.** New script `scripts/generate_source_concept_map.py` drafts customer source-to-concept mappings via OHDSI's `concept_relationship` Maps-to chain, producing a `source_to_concept_map`-shaped output table plus a markdown coverage report at `reports/source_mapping_coverage_<timestamp>.md`. The generator is opt-in per table via a new `source_vocabulary[]` field in per-table YAML schemas (back-compat preserved — existing projects with no `source_vocabulary` configured are unaffected). Designed for customers landing on STCM bootstrap as the most common adoption path; the generator drafts mappings, the customer reviews and refines, and the resulting STCM table feeds the runtime resolver. See runbook Section 6.3 for the end-to-end workflow.

**Customer-visible behavior change — vocabulary loader fails loudly on empty loads.** Before v2.0.7, `templates/project_scaffold/src/01_load_vocabulary.py` could complete successfully even when one or more required vocabulary tables (`concept`, `concept_relationship`, `vocabulary`, `domain`) ended up empty after the load — silently passing to subsequent pipeline steps that would then fail with cryptic downstream errors. v2.0.7 adds a post-load assertion that counts rows in each required table and raises `RuntimeError` listing the empty tables plus a three-step troubleshooting flow (CSV presence check, COPY INTO type-mismatch surface, volume path verification). See runbook Section 6.1 for the full troubleshooting workflow. Customers re-running existing projects after upgrading will see this assertion fire only if their previous load was already silently broken; well-loaded projects are unaffected.

**Hardening — scaffolder no longer ships Python bytecode to customer projects.** Before v2.0.7, `scripts/scaffold_omop_project.py`'s template-tree walk would propagate any `__pycache__/` directories or `*.pyc` files present in the dev-side `templates/project_scaffold/` tree into the customer's scaffolded project. Customers ended up with stale bytecode tracked in their fresh projects. v2.0.7 explicitly excludes `__pycache__/` directories and `*.pyc` files from the scaffolder's tree walk; bytecode now stays dev-side only.

**Documentation expansion.**

- **New runbook Section 6.3** (`templates/project_scaffold/docs/omop-runbook.md`): "Drafting `source_to_concept_map` at scale" — end-to-end workflow for the new mapping generator, including invocation patterns, per-vocabulary configuration shape, coverage report interpretation, and the customer-review handoff.
- **New runbook Section 6.4**: "Vocabulary refresh procedure" — documents the pattern customers should follow when the OHDSI concept tables update (typically quarterly), including the right-shape of running `01_load_vocabulary.py` with a fresh CSV drop, post-load STCM regeneration considerations, and the validator's role in catching pre-existing data that no longer maps cleanly under refreshed vocabularies.
- **New runbook Section 9**: "Security considerations" — six skill-relevant items covering PHI handling in source-to-concept mapping work, Unity Catalog grants for mapping output tables, audit logging for STCM modifications, vocabulary loader's read-from-Volume privilege model, the validator's read-only stance, and the scaffolder's idempotency guarantees.
- **New architectural decision document `references/AD-002-skill-drafts-customer-reviews.md`** (sibling to AD-001): formalizes the "skill drafts; customer reviews and refines" handoff pattern, with the source-to-concept mapping generator as the worked example and broader applicability to future generators (e.g., FK validation rules, domain-classification overrides, cross-table reconciliation rules).
- **New per-table YAML schema field `source_vocabulary[]`**: opt-in metadata field added to per-table YAML configs that the new mapping generator consumes. Schema validation extended via the existing Pydantic schema layer; existing per-table YAMLs without `source_vocabulary` validate cleanly under the new schema.

**Discipline (visible to anyone reading the repo's customer-facing artifacts).** v2.0.7 is the first release where the dev-repo / public-repo sync applies a comprehensive scrub of internal engineering vocabulary across customer-shipped surfaces (internal labels in module docstrings, internal finding IDs in code comments, forward-looking internal-plan references). Customers reading scripts or runbook prose will not encounter dev-repo engineering vocabulary. The customer-scaffolded project's `templates/project_scaffold/docs/CHANGELOG.md` continues the established convention from v2.0.5 / v2.0.6 of naming behavior-change finding IDs in section headers as cycle-of-record markers; those finding IDs are intentional and customer-relevant (each one corresponds to a behavior change documented in the runbook).
